## 在使用大数据系统十多年后，数据专业人员在即将到来的一年面临一系列新的挑战。

为世界上大多数数据管道提供燃料的开源项目 Apache Spark 到 2020 年已经 10 岁了。为了庆祝这一里程碑，超过 50，000 名数据科学家、数据工程师、分析师、商业领袖和其他数据专业人士观看了由 Databricks 组织的年度 Spark + AI 峰会的不同会议。

如果这告诉了我们什么，那就是大数据不再是一组新兴的利基技术。在过去的十年里，大数据缓慢但稳步地成为主流:各种规模和行业的公司都采用了这些技术，旨在利用自己掌握的大量数据。

> **过去十年，大数据缓慢但稳步地成为主流。**

换句话说，大数据变成了数据。随着使用必要工具的成本和复杂性的下降(由于云提供商和开源项目的成熟)，拥有大量数据并使用分布式系统来管理它们现在是一种规范。

在这种环境下，处理大数据不再是一种挑战，而是一种必然，过去几年出现了三个主要挑战，它们肯定会成为 2021 年数据社区的话题。

# 1.元数据管理:在数据迷宫中导航

每个人采用大数据技术和构建数据湖的速度有其副作用。当然，我们现在有大量的数据和强大的数据处理管道，但是如何回答像*“谁拥有这个数据集？”*，*“这个数据集的血统是什么？”*，更重要的是*“在哪里可以找到符合我需求的数据？”。*

存储和处理数据不再是一项挑战，但现在下一步是真正实现对所有这些数据的民主化访问，并提供搜索功能，使公司内的不同用户能够在他们可以支配的多个数据湖、数据存储和数据库中导航。

实现这个目标的关键是**元数据**。通过存储关于每个数据集的特定信息(一些例子:描述、标签、语义、所有权和模式演变)，我们确保用户能够以优化的方式利用数据，而无需外部帮助。

在过去几年中，有两个开源工具旨在解决这一特定挑战，它们是:

## 阿蒙森

早在 2019 年，Lyft 就首次开源了 Amundsen，它是根据*“谷歌数据搜索”*的想法建造的。它支持不同的数据目录，如 Apache Atlas，并提供多种功能，如索引不同的数据资源(表、仪表板、流等)。)和支持页面分级式搜索(基于使用模式)。Amundsen 依赖于基于拉的方法，这意味着工具本身从不同的资源中提取元数据。

## [数据中心](https://engineering.linkedin.com/blog/2019/data-hub)

与 Amundsen 不同，data hub(LinkedIn 于 2019 年开源)采取了基于事件的方法，旨在解决数据发现挑战。它支持元数据模型的不断发展，并依赖于元数据提供者通过基于流的 API 推送更新。

# 2.数据湖 2.0:为数据湖增加便利

在构建和使用数据湖的最初几年中出现的第二个挑战是维护它们有多复杂。

无论是 Hadoop 集群还是 S3 桶，数据湖都需要在幕后付出大量努力来提供基本的数据库功能，如模式进化、ACID 事务和版本控制。

当然，大数据从一开始就没有向我们承诺过这样的能力，但是在最初的数据湖设计工作几年后，它变得非常不切实际，因为多种常见需求(例如模式演变)最终会出现，并且必须得到满足。

为了解决这些问题，过去几年中引入了多种工具来抽象管理数据湖和在其上构建表的复杂性。在这些工具中，我们可以提到:

## [三角洲湖泊](https://delta.io/)

在首次向 Databricks 客户提供后，Delta Lake 于 2019 年由该公司开源，提供了一系列功能和功能，使数据湖更加可靠和方便。

举几个例子，Delta Lake 确保了数据湖之上的 ACID 事务、数据版本和模式演化。

## [阿帕奇冰山](https://iceberg.apache.org/)

冰山由网飞在 2018 年开源，然后捐赠给阿帕奇软件基金会，它引入了一系列新概念，改变了大数据表的组织方式(例如，它跟踪文件而不是目录/分区)。

它在很大程度上依赖于每个表具有不同的元数据属性，并有效地增强了大型表的查询性能。就像三角洲湖一样，冰山提供了多种优势，如模式进化和时间旅行。

# 3.数据网格:回到绘图板

尽管在大数据生态系统中做出了令人印象深刻的持续努力，但对于一些公司和用例来说，在大数据的最初几年后，正确的举措可能是以不同的方式做事。

[数据网格](https://martinfowler.com/articles/data-monolith-to-mesh.html)，ThoughtWorks 的 Zhamak Dehghani 在 2019 年推出的一种新范式，提供了一套管理大数据系统的新概念。它要求我们抛弃依赖于集中化的单一数据湖设计，转而支持具有面向领域愿景的分布式架构。

该范式依赖于由专门团队管理的自助式数据基础架构(与当前基于数据湖的设计中使用的工具集相同)的概念，而每个领域数据集被视为一个独立的产品。

考虑到这些挑战，未来一年肯定会在数据领域带来更多创新，让数据专业人员更方便地充分利用他们的数据集。
